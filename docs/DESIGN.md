# 设计说明

## 文档目的

本文用于说明为什么项目需要独立 Mock Service，以及它和真实微服务后端在职责上的边界。公开展示时，如果不把这部分讲清楚，读者很容易把 Mock 仓库误解为“简化版后端”。

## 仓库定位

`xiaotuxian-mall-mock-service` 是一个独立运行的本地接口服务。它的主要目标是为 PC 商城前端提供可控、稳定、低成本的演示接口，而不是承载真实交易后端的全部职责。

## 为什么需要 Mock Service

在公开展示场景下，前端页面经常需要：

- 快速启动
- 可重复展示
- 不依赖外部中间件
- 在本地即可完成页面联调

如果每次演示都要求同时启动 MySQL、Redis、RabbitMQ 和多套微服务，演示成本会显著升高，也更容易受环境波动影响。Mock Service 的意义，就是为这些页面能力提供一个稳定入口。

## 适用场景

Mock Service 主要适用于：

- 首页、分类、详情、购物车、结算等页面联调
- 权益字段展示验证
- 页面截图制作
- 演示流程回归
- 没有完整基础设施时的本地运行

它并不负责演示真实微服务的注册发现、消息驱动和持久化链路。

## 不依赖哪些组件

该仓库明确不依赖以下基础设施：

- MySQL
- Redis
- RabbitMQ
- Nacos
- OpenFeign
- 网关
- 真实订单消费链路

这也是它和主仓真实微服务链路最核心的区别。

## 数据来源设计

Mock Service 的基础数据来自 `src/main/resources/mock` 下的 JSON 文件。这些数据文件提供：

- 首页分类与 banner
- 商品详情与 SKU
- 购物车、结算、地址、订单示例
- 优惠券与礼品卡演示数据

这套数据结构的目标不是完整建模所有业务，而是与前端现有页面字段保持兼容。

## 运行时数据设计

项目除基础 JSON 外，还保留 `data/` 目录作为运行时变更数据区。它用于保存本地演示过程中的可变状态，例如：

- 购物车变更
- 地址变更
- 订单提交
- 优惠券使用
- 礼品卡绑定或扣减

这样可以在多次演示过程中保留用户可见状态，同时又能通过删除运行时文件快速重置。

## 与真实微服务的关系

Mock Service 与真实微服务不是主从关系，而是“用途不同的两套演示支撑”：

- 真实微服务：用于解释库存、订单、支付和异步链路
- Mock Service：用于快速跑通前端页面和稳定制作公开截图

因此 Mock 仓库文档必须避免写成“它也用了 MySQL / Redis / RabbitMQ”的样子。

## 字段兼容设计

Mock Service 在字段层面尽量与前端当前页面保持兼容，重点覆盖：

- 商品基本信息
- SKU 库存
- 购物车项数量
- 结算页地址与权益字段
- 订单详情展示字段

这种兼容策略的目的是让前端在 Mock 模式下无需修改页面结构就能展示核心流程。

## 演示稳定性优先

与真实微服务相比，Mock Service 的设计优先级更偏向于：

- 返回结构稳定
- 页面展示可控
- 数据可快速修复
- 截图效果统一

因此它允许通过静态 JSON 和少量运行时数据文件来换取更高的演示可用性。

## 为什么单独开仓

将 Mock Service 单独拆成公开仓库有两个好处：

1. 前端读者可以单独查看本地联调依赖
2. 平台主仓可以更聚焦真实微服务与整体能力，不必把 Mock 技术栈和真实后端混在一起

这样做能让三个仓库的职责边界更清晰：

- 主仓：平台总览与真实微服务
- Mock 仓：本地联调接口服务
- 前端仓：用户侧页面与交互流程

## 当前设计边界

Mock Service 当前并不追求：

- 服务治理能力
- 高并发吞吐能力
- 生产级消息处理
- 真实库存预扣实现

它追求的是“公开展示足够真实，联调体验足够稳定”。

## 公开展示价值

对于简历、面试或作品集展示来说，Mock Service 的价值在于：

- 可以独立讲清联调支撑能力
- 可以解释为什么项目保留双模式
- 可以支撑页面截图和中文公开文档
- 可以降低外部读者理解项目的门槛

因此，Mock Service 不是“弱化版后端”，而是“公开展示和本地联调的专用支撑层”。
