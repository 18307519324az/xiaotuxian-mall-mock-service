# 接口范围说明

## 核心接口范围

本地 Mock Service 主要提供以下演示接口：

- 认证与用户：`/login`、`/register`、`/member/profile`
- 首页与内容：`/home/*`、`/category/*`、`/goods`、`/brand/*`、`/topic/*`
- 购物车与结算：`/member/cart`、`/member/order/pre`
- 订单流程：`/member/order`、`/member/order/repurchase`、`/member/order/{id}`
- 权益能力：`/member/coupon`、`/member/coupon/exchange`、`/member/gift-card`、`/member/gift-card/bind`
- 用户侧能力：评价、积分、消息、邀请、售后、客服等接口

## 返回结构

接口返回结构与商城前端保持兼容，包含以下字段：

- `code`
- `msg`
- `traceId`
- `result`

## 使用定位

- 适合本地页面演示
- 适合前端接口兼容校验
- 适合在不启动数据库和中间件时做联调

该服务面向本地演示与前端兼容验证，不作为生产服务使用。

## 主要请求特点

Mock Service 的接口设计重点不是复杂鉴权，而是保证前端当前页面能稳定读取到兼容字段。对公开展示最重要的价值包括：

- 路由路径与前端当前调用保持一致
- 返回结构统一
- 可重复返回演示数据
- 局部状态可在运行时写入 `data/` 目录

## 参数与返回值风格

大部分接口会采用以下风格：

- 查询参数或路径参数尽量与前端现有调用一致
- 返回对象统一包装在 `result` 字段中
- 保留 `code`、`msg`、`traceId` 等结构，降低前端改造成本

这样前端在 Mock 模式和真实服务模式之间切换时，不需要为每个页面单独改造渲染逻辑。

## 接口分类的作用

将接口按首页、分类、商品、购物车、结算、订单、权益、用户侧能力进行分类，有两个目的：

1. 让读者快速知道 Mock Service 覆盖了哪些页面
2. 让前端联调时能够定位某一组接口的边界

这也是公开仓库里保留 `API.md` 的原因。

## 不负责的能力

需要明确的是，Mock Service 不负责：

- Redis 预扣库存
- RabbitMQ 异步下单
- MySQL 事务落库
- Nacos 服务治理

这些属于真实微服务模式的职责，不应写进 Mock 仓的技术栈或接口说明。

## 联调建议

在实际联调中，建议优先验证：

- 首页和分类接口是否能稳定返回
- 商品详情与库存字段是否齐全
- 购物车和结算接口是否带上权益字段
- 订单详情结构是否满足页面回显需要

这样可以更快发现是页面问题还是接口兼容问题。

## 公开展示价值

对外展示时保留这份 `API.md`，可以让读者快速知道：

- Mock Service 覆盖了哪些页面能力
- 它的接口边界在哪里
- 它为什么适合作为本地联调支撑层

这比只在 README 里写一句“支持联调”要更具体。
