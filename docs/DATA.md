# 数据说明

## 静态演示数据

基础 JSON 样例位于 `src/main/resources/mock`。

## 运行时数据

运行过程中的可变 JSON 文件会写入 `data/` 目录。

示例包括：

- 购物车运行时快照
- 订单运行时快照
- 地址运行时快照
- 优惠券与礼品卡运行时快照

## 持久化模型

- 静态样例文件用于初始化演示状态
- 运行时文件记录用户在本地演示中的可见变更
- 删除运行时 JSON 文件后，可以将可变演示状态恢复到初始状态

## 为什么要拆成静态与运行时两层

如果所有数据都只放在一个静态 JSON 文件里，前端一旦发生下单、改地址、绑定礼品卡等操作，就无法保留演示结果。拆成两层后：

- 静态层负责提供基础样例
- 运行时层负责记录当前会话中的可变状态

这能明显提升本地演示的可重复性。

## 常见重置场景

以下场景通常需要清理运行时数据：

- 重做首页和结算页截图
- 重新演示购物车和订单流程
- 需要恢复到初始优惠券或礼品卡状态
- 多轮联调后数据混乱

删除 `data/` 下相关 JSON 文件后，即可恢复到基础演示状态。

## 编码与文本质量

由于公开截图和页面渲染依赖这些 JSON 数据，数据文件必须保持 UTF-8 且不能混入乱码字符。否则即使页面逻辑没问题，README 截图也会出现不专业的展示效果。

## 推荐维护方式

对于公开展示使用的 JSON 数据，建议维护时遵循以下原则：

- 优先修正源数据，不用图片掩盖问题
- 文本字段统一保存为 UTF-8
- 演示数据尽量选择可重复使用的样例
- 运行时数据与基础样例分开存放

这样能降低截图、联调和回归验证的维护成本。

## 公开展示中的作用

数据说明文档对公开仓库的意义在于，它能帮助读者理解：

- 为什么 Mock Service 不需要数据库
- 为什么页面状态仍然可以发生变化
- 为什么某些截图或演示结果可以稳定复现

这使 Mock 仓的设计意图更加清晰。
